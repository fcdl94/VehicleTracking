Vehicle Tracking System Documentation

Project specification

The aim of this project is to design and implement a RESTful web service that can track the presence of vehicles in an area with restricted access and, based on the tracking information, can provide permission to new vehicles to enter the area and suggestions about the path to their destinations.

The service uses a model of the area that is a graph, with nodes representing roads and parking areas, and arcs representing the possibility to go from one node to another one. When a new vehicle wants to enter the area, it has to ask the service for permission, specifying both the entry point (i.e. its current position) and the desired destination in the area. The service will check if it is possible to allow the entrance of the new vehicle, based on the area model (which can include a number of constraints, such as capacity of parking areas and of roads) and based on the current and expected future positions of the other vehicles in the area. If the service allows entrance, it responds by also indicating a suggested path that the new vehicle is expected to follow from the entry point to the destination, and a unique identifier assigned by the service to the new vehicle. While the vehicle is moving to its destination it will periodically send information about its current position to the service, which will update its tracking information. If the service realizes that a vehicle is not following the suggested path, it will compute an updated path that starts from the current vehicle position and will send it back to the vehicle in response to the position tracking message. The area model must be specified in an XML document, for which a schema has to be designed. The service must upload the area model from the XML file at startup and it must allow administrators to collect various kinds of information about the vehicles currently in the area and their expected routes.

Model

The model is made by two main entities: vehicles and graph.

The graph represents the list of nodes and their connections. A node can be a road or a parking area and it can be crossed by a vehicle. Every node can have also many ports, that are entry point into that node (think about a parking area, there can be many entry points where there is a barrier). The connection is a link between two nodes, in particular, between two ports from different nodes.

As said, a node can be a parking area or a road. It is important that every node have constraints, but a general node have no specified constraint while the derivation (road, parking area) can define their own.

The vehicle entity represents a vehicle in the system. It has many information as the plate number, the entry point, the time of the last update of the position that are relevant for the administration. It has also information about the status: position, destination and state (parked, in transit). Those are useful to compute the path, that is an ordered list of connection that the vehicle must cross to reach the destination.

The server

The server was developed by implementing three layers.

         The resource layer, the one that deal with Tomcat and the client calls.

         The application layer, the one really implementing the logic of the system. It is able to create new vehicles, compute path, update path and so on. This is the most important part of the system, because it is here that the system lives.

         The model layer in which there are the annotated class for JAXB translation. The classes here are automatically made by the XJC script from the schema.

Concurrency

Considering concurrency Issues, I had to insert some synchronized sections.

For the creation of vehicles, the main problem is related to path computation. In fact, if two vehicles compute together the path, they may violate constraints on some node. So, I synchronized the operation of path computation, that happens in GraphApp class.

To prevent any issue on vehicles, all the operation on a vehicle object are synchronized (gets, update, update position and delete). This means that also the path, that is related to one and only one vehicle, is modified in synchronized way.

To prevent issue in read operations for VTService and for the Nodes, all the maps, lists and sets use synchronized implementations

Vehicle Client

I have developed an example client. This client simulates the behavior of a real user. The client first creates the new vehicle with a POST on vehicle, then it POST the position following the path until the destination.

Reached the destination, it performs a PUT on the vehicle, updating the information (current position, new destination) to reach an end point of the system. Then, it follows the path and, arrived at destination, it ask to exit with the DELETE operation.

Administrator Client

Administrator client is quite simple, it only performs a get on vehicles. This is made to get all the vehicle in the system, as the requirements ask.
